# Demo alerts

## Dashboard

Demo alerts can be observed through the following investigation surfaces:

- the **demo app** traffic generator
- the **Application alerts dashboard**
- the **Alertmanager preview** on the demo page
- the **Loki button logs** preview

The **Application alerts dashboard** shows the key application signals that change when demo endpoints and demo scenarios are triggered.

![Application alerts dashboard](../images/application-alerts.png)

The dashboard includes panels for:

- demo application health
- HTTP 5xx error rate
- inflight requests
- P95 request latency
- HTTP status distribution

These panels allow operators to observe how the monitoring system reacts when demo alerts are triggered.

---

## Purpose

Demo alerts are intentionally generated to validate the monitoring pipeline end to end.

They are used to test:

- Prometheus metrics
- alert rule evaluation
- Alertmanager routing
- dashboard visibility
- runbook procedures
- public demo investigation flow

Demo alerts are useful for validation and demonstrations, but they are not meant to behave exactly like production alerts.

---

## Demo alert model

The demo flow now has **two layers** of alert behavior.

### 1. Fast demo alerts

These fire quickly after demo traffic is generated.

They exist mainly to confirm that:

- metrics changed
- Prometheus evaluated the rule
- Alertmanager received the alert
- the UI can display the alert state clearly

Fast demo alerts:

- `DemoAppButtonError503`
- `DemoAppButtonSlow`

### 2. Longer warning alerts

These are normal alert-style conditions based on sustained signal over time.

They may appear later than the fast demo alerts, depending on:

- traffic volume
- burst count
- latency value
- rule thresholds
- `for:` windows

Related longer alerts:

- `DemoAppHigh5xxRate`
- `DemoAppHighP95Latency`

These are documented in the application alerts runbook.

---

## Routing behavior

**Important**

Fast demo alerts are routed to a **blackhole receiver** in Alertmanager.

They do **not generate real operator notifications**.

This keeps the public demo safe and prevents noise while still allowing the full monitoring path to be observed.

Monitoring pipeline for demo alerts:

```
demo endpoint / scenario
     │
     ▼
Prometheus metrics
     │
     ▼
alert rule evaluation
     │
     ▼
Alertmanager routing
     │
     ├── production and warning alerts → Telegram / downstream routing
     └── demo alerts → blackhole
```

This means demo alerts should still be visible in:

- Prometheus
- Alertmanager preview
- demo app observability widgets
- dashboards

But they should not page or notify real operators.

## Investigation entry points

When validating demo alerts start with:

**1. demo app**
- trigger a scenario
- observe summary state
- review recent client-side events

**2. Alertmanager preview**
- confirm alert visibility
- confirm alert name and state

**3. Loki button logs**
- confirm forced_error or forced_slow events

**4. Application alerts dashboard**
- confirm that traffic affected the expected application signals

## DemoAppButtonError503

**Severity:** demo

### Description

This alert is generated by intentionally returning HTTP 503 responses from the demo application.

It is the fast demo alert used to confirm that an error scenario reached Alertmanager.

This alert is separate from the longer application warning alert DemoAppHigh5xxRate.

### Trigger source

Endpoint:

```
/error?code=503
```

Example:

```
https://demo.142.93.143.228.nip.io/error?code=503
```

It can also be triggered via the demo UI:

- single /error 503
- trigger 5xx demo
- trigger combined demo

Expected behaviour

1.	request returns HTTP 503
2.	app_forced_errors_total increases
3.	the fast demo alert becomes visible in Alertmanager
4.	the alert is routed to the blackhole receiver
5.	the demo page summary updates to show detection

### What to verify

Check button logs:

```
curl -fsS "https://demo.142.93.143.228.nip.io/_obs/logs?mode=buttons&limit=20"
```

Check alert preview:

```
curl -fsS "https://demo.142.93.143.228.nip.io/_obs/alerts"
```

Check Prometheus metric:

```
increase(app_forced_errors_total{service="demo-app",code="503"}[1m])
```

### Notes

This alert is intended to appear quickly and give fast feedback in the demo UI.

It does **not** mean the sustained 5xx warning rule has fired.

## Scenario behaviour

The demo UI supports several scenario types.

### 1. 5xx demo

Used to quickly produce error traffic.

Expected result:

- DemoAppButtonError503 appears first
- DemoAppHigh5xxRate may appear later if the rule threshold is satisfied

### 2. Latency demo

Used to quickly produce slow request traffic.

Expected result:

- DemoAppButtonSlow appears first
- DemoAppHighP95Latency may appear later if the rule threshold is satisfied

### 3. Combined demo

Used to mix error and slow traffic in a single validation flow.

Expected result:

- one fast demo alert may appear before the other
- the UI may show partial success first
- slower warning-style alerts may require more evaluation time

This behavior is normal.

## Troubleshooting demo scenarios

### Scenario traffic is visible in logs but no demo alert appears

Possible causes:

- Prometheus has not evaluated the rule yet
- Alertmanager preview has not refreshed yet
- the rule window has not been crossed yet
- insufficient request volume

Check:

- button logs
- alert preview
- current scenario counters in the demo UI

### Fast demo alert appears but longer warning alert does not

This is expected in many cases.

Longer warning alerts depend on sustained signal and alert thresholds.

Check:

- burst count
- interval
- latency value
- alert rule thresholds
- `for:` window duration

### Demo alert disappeared after appearing briefly

This can happen when the short-lived demo signal leaves the evaluation window.

The alert was still useful if it confirmed that the pipeline worked.

Check:

- recent logs
- recent UI summary
- dashboard signals around the same time

### Alert did not notify Telegram

This is expected for fast demo alerts.

They are intentionally routed to the **blackhole receiver**.

## Resolution criteria

A demo scenario is considered successful when:

- the expected button logs are visible
- the expected fast demo alert appears in Alertmanager preview
- dashboard signals reflect the generated traffic
- the public demo UI summary reflects the expected state
