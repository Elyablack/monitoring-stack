# Observability Runbook

This document describes how to diagnose and resolve alerts generated by the monitoring stack.

It is intended for operators responding to monitoring alerts.

---

# Architecture Overview

The monitoring stack consists of the following components:

| Component | Purpose |
|--------|--------|
Prometheus | Metrics collection and storage |
Alertmanager | Alert routing and grouping |
Loki | Log storage |
Promtail | Log shipping agent |
Grafana | Visualization |
tg-relay | Telegram alert delivery |
demo-app | Example instrumented application |

# Quick Monitoring Health Check

Before investigating any alert, ensure the monitoring stack itself is healthy.

Run:

```bash
monitor

This command verifies:
	•	container status
	•	Prometheus readiness
	•	Alertmanager readiness
	•	Loki readiness
	•	Prometheus targets
	•	firing alerts
	•	pending alerts
	•	tg-relay status
	•	demo-app health
	•	recent Promtail errors

If the monitoring stack is unhealthy, resolve that first.

⸻

Alert Severity Levels

Severity	Meaning	Response
critical	Service outage or data loss risk	Immediate investigation
warning	Potential degradation	Investigate soon
info	Informational	Monitor


⸻

Alerts

⸻

Alert: NodeExporterDown

Severity: critical

Description

Prometheus cannot scrape node_exporter metrics.

Possible causes:
	•	host unreachable
	•	exporter stopped
	•	network problem
	•	configuration error

⸻

Investigation

Check host connectivity

ping <host>

Attempt SSH:

ssh <host>

If unreachable:
	•	verify network
	•	check VPS status
	•	verify Tailscale
	•	check for recent reboot

⸻

Check exporter endpoint

curl http://<host>:9100/metrics

If not reachable:

systemctl status node_exporter

If containerized:

docker ps
docker logs <node-exporter>


⸻

Check Prometheus target

Open Prometheus UI:

Status → Targets

Inspect scrape errors.

⸻

Alert: HostLowDiskSpace

Severity: critical

Description

Filesystem free space below 10%.

⸻

Investigation

Identify filesystem:

df -h

Find large directories:

du -sh /* | sort -h

Or check logs:

du -sh /var/log/*

Inspect Docker usage:

docker system df

List resources:

docker images
docker volumes
docker containers


⸻

Resolution

Possible actions:
	•	remove unused Docker images
	•	rotate logs
	•	remove temporary files
	•	delete old backups

Do not remove:
	•	database volumes
	•	active backups
	•	required application logs

⸻

Alert: HostMemoryPressure

Severity: warning

Description

Available system memory is critically low.

⸻

Investigation

Check memory usage:

free -h

Identify memory-heavy processes:

ps aux --sort=-%mem | head

Check for OOM events:

dmesg | grep -i oom

or

journalctl -k | grep -i oom

Inspect containers:

docker stats


⸻

Alert: HostHighCpuLoad

Severity: warning

Investigation

Check CPU usage:

top

or

htop

Identify CPU-heavy processes:

ps aux --sort=-%cpu | head

Check load averages:

uptime

Inspect containers:

docker stats


⸻

Alert: HostRebootDetected

Severity: warning

Description

Detected change in:

node_boot_time_seconds


⸻

Investigation

Check uptime:

uptime

Check reboot history:

last reboot

Inspect previous boot logs:

journalctl -b -1

Verify services restarted:

docker ps


⸻

Alert: DemoAppDown

Severity: critical

Description

Prometheus cannot scrape demo-app metrics.

⸻

Investigation

Check container:

docker compose ps demo-app

Check health endpoint:

curl http://demo-app:8081/healthz

Check metrics endpoint:

curl http://demo-app:8081/metrics

Inspect logs:

docker logs --tail 200 demo-app


⸻

Alert: DemoAppHigh5xxRate

Severity: warning

Investigation

Check 5xx error rate:

sum(rate(http_requests_total{service="demo-app",status=~"5.."}[2m]))

Inspect status distribution:

sum by (status) (rate(http_requests_total{service="demo-app"}[2m]))

Identify failing endpoints:

sum by (path,status) (rate(http_requests_total{service="demo-app"}[2m]))

Check logs in Loki:

{service_name="demo-app"} | json | status >= 500


⸻

Alert: DemoAppHighP95Latency

Severity: warning

Check latency:

histogram_quantile(
  0.95,
  sum by (le)(
    rate(http_request_duration_seconds_bucket{service="demo-app"}[5m])
  )
)

Check inflight requests:

http_inflight_requests{service="demo-app"}

Check slow logs:

{service_name="demo-app"} | json | duration_ms > 500


⸻

Alert: DemoAppHighInflight

Severity: warning

Check inflight requests:

http_inflight_requests{service="demo-app"}

Check traffic rate:

sum(rate(http_requests_total{service="demo-app"}[1m]))

Incident Checklist

When handling an alert:
	1.	Confirm alert validity
	2.	Check monitoring stack health
	3.	Identify affected service
	4.	Check metrics and logs
	5.	Identify root cause
	6.	Apply fix
	7.	Verify alert cleared
	8.	Document incident
