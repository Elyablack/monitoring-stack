# Observability Runbook

## Purpose

This runbook provides investigation and remediation steps for alerts generated by the monitoring stack.

It is intended for operators responding to monitoring alerts and validating that the stack is functioning correctly end to end.

Use this runbook to:

- confirm alert validity
- verify monitoring stack health
- investigate metrics, logs, and dashboards
- identify the affected component
- apply remediation
- verify alert resolution

## Architecture overview

The monitoring stack consists of the following components.

- **Prometheus** — metrics collection and storage
- **Alertmanager** — alert routing and grouping
- **Loki** — log storage
- **Promtail** — log shipping agent
- **Grafana** — visualization
- **tg-relay** — Telegram alert delivery
- **demo-app** — example instrumented application
- **control-plane** — optional alert-driven orchestration and remediation surface

---

## Monitoring architecture

```
demo-app
   │
   │ metrics
   ▼
Prometheus
   │
   │ alerts
   ▼
Alertmanager
   │
   │ notifications
   ▼
tg-relay → Telegram

Logs pipeline

demo-app → Promtail → Loki → Grafana
```

If control-plane integration is enabled, selected alerts may also continue into an alert-driven orchestration workflow.

```
Alertmanager → control-plane → decision → task → run
```

## Alert handling workflow

```
Alert fires
     │
     ▼
Check monitoring health
     │
     ▼
Identify affected component
     │
     ▼
Investigate metrics and logs
     │
     ▼
Apply remediation
     │
     ▼
Verify alert resolution
```

Standard response flow:

- confirm alert validity
- check monitoring stack health
- identify affected component
- investigate metrics and logs
- apply remediation
- verify alert resolved

If the alert is expected to trigger downstream automation, also verify control-plane activity.

- **Control-plane** — read-only alert-to-task-to-run investigation view

- use **Grafana** for deeper internal investigation
- use **category runbooks** for alert-specific remediation steps

## Quick monitoring health check

Before investigating any alert ensure the monitoring stack itself is healthy.

Run:

```
monitor
```

This command checks:

- container status
- Prometheus readiness
- Alertmanager readiness
- Loki readiness
- Prometheus targets
- firing alerts
- pending alerts
- tg-relay health
- demo-app health endpoint
- recent Promtail errors

If the monitoring stack itself is broken, fix it first.

---

## Quick triage

When an alert fires quickly check:

- Is the monitoring stack healthy?
- Is the alert still firing?
- Is the host reachable?
- Did the host reboot recently?
- Is the service or container running?
- Are metrics still being scraped?
- Are related logs still arriving?
- Is the issue visible in dashboards?
- If automation is expected, did control-plane receive and process the alert?

## Alert severity levels
	
- **critical**
  - service outage
  - immediate investigation required
- **warning**
  - degradation risk
	- investigate soon
- **info**
  - informational
	- monitor only
- **demo**
  - demo-only scenario
	- useful for validation and testing
  - may be intentionally routed differently from production-facing alerts

## Incident checklist

When handling an alert:

- confirm alert validity
- check monitoring stack health
- identify affected component
- check metrics and logs
- review dashboards
- identify root cause
- apply fix
- verify alert cleared
- document incident

If control-plane is involved, also confirm:

- alert matched expected routing
- decision was created
- task was queued
- run completed or failed as expected

---

## Incident documentation

When resolving incidents record:

- alert name
- severity
- time detected
- affected host or service
- root cause
- remediation applied
- verification steps
- prevention steps

If orchestration was involved also record:

- whether the alert reached control-plane
- decision outcome
- task status
- run result

## Useful commands

Check monitoring stack:

```
monitor
```

Check Prometheus alerts:

```
curl http://127.0.0.1:9090/api/v1/alerts | jq
```

Check Prometheus targets:

```
curl http://127.0.0.1:9090/api/v1/targets | jq
```

Check Loki health:

```
curl http://127.0.0.1:3100/ready
```

Check Alertmanager health:

```
curl http://127.0.0.1:9093/-/ready
```

Check container status:

```
docker compose ps
```
Check container logs:

```
docker compose logs --tail=100
```

Check demo app health:

```
curl -fsS http://127.0.0.1:8080/healthz
```

Check control-plane summary when enabled:

```
curl -fsS http://127.0.0.1:8081/api/control-plane/summary?window=24h | jq
```

Check control-plane decisions when enabled:

```
curl -fsS http://127.0.0.1:8081/api/control-plane/decisions?window=24h&limit=10 | jq
```

Check control-plane tasks when enabled:

```
curl -fsS http://127.0.0.1:8081/api/control-plane/tasks?window=24h&limit=10 | jq
```

Check control-plane runs when enabled:

```
curl -fsS http://127.0.0.1:8081/api/control-plane/runs?window=24h&limit=10 | jq
```
